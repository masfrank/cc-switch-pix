# CC Switch v3.17.0

> This release focuses on **eliminating the pain of re-entering provider configurations across multiple apps** — bringing cross-app reuse to the "models & providers" configuration. Three core additions: first, **cross-app quick copy**, which lets you distribute a provider's config (API Key / Base URL / name / notes / icon, etc.) to other apps with one click from any provider card, automatically converting the config format for each target app during the copy; second, **provider config import/export**, supporting batch export of selected providers to a JSON file and import from a file or clipboard for backup and cross-device migration; third, **app-level permission control for Universal Providers**, upgrading the simple boolean enable flag to a configurable `enabled` permission object, fully backward-compatible with existing configs.

## Usage Guides

The new capabilities in this release are spread across the provider list, the Universal Provider panel, and settings. The following scenarios are worth reading alongside:

- **Cross-app copy**: click the new "Copy to other apps" button (share icon) on any provider card, then check the target apps. CC Switch extracts the API Key and Base URL from the source provider and regenerates the config in each target app's structure (Claude's `env.ANTHROPIC_*`, Codex's `auth` + TOML `config`, Gemini's `env.GEMINI_*`, OpenCode's `options`, OpenClaw / Hermes flat structures) — no more per-app re-entry.
- **Import / Export**: a new "Import/Export" button sits at the top of the provider list. Export lets you multi-select providers and auto-downloads a dated JSON file; import accepts a local JSON file or pasted content, with validation on file size, JSON validity, and version.
- **Universal Provider permissions**: the `apps` field is upgraded from a boolean to an object that allows fine-grained control of each app's enabled state, fully backward-compatible with existing configs.

## Overview

CC Switch v3.17.0 is a feature update following v3.16.5, centered on **decoupling "models & providers" configuration from specific apps to improve cross-app efficiency and consistency**. Before this, even for the same model provider and model list, users had to configure each app separately — Claude Code, Codex, Gemini, OpenCode, OpenClaw, and Hermes. This release provides two layers of solution for this problem: **quick copy** (cross-app config cloning + format conversion) and **import/export** (backup and migration), plus a permission-dimension enhancement to the existing **Universal Provider**.

Cross-app copy is not a simple field copy — each app's `settingsConfig` structure differs significantly (Claude uses `env.ANTHROPIC_BASE_URL` / `ANTHROPIC_AUTH_TOKEN`, Codex uses `auth.OPENAI_API_KEY` plus a TOML `config` string, Gemini uses `env.GEMINI_*`, OpenCode nests under `options`, OpenClaw and Hermes use flat camelCase / snake_case). CC Switch resolves `base_url` + `api_key` as an intermediate credential pair from the source provider, then reassembles them into the target app's structure while preserving display metadata like name, notes, icon, and website. Official / OAuth providers (without base_url and api_key) are rejected with a reason, avoiding bad configs.

## Added

### Cross-app quick copy of providers

This is the core feature of this release, addressing the "same provider must be configured per app" feedback in issue #4599. On any app's provider card, clicking the new "Copy to other apps" button (Share2 icon) in the action area opens an app-selection dialog where you can select-all or check target apps. CC Switch will:

1. Resolve `base_url` + `api_key` from the source provider (reusing the existing `resolve_usage_credentials`, which covers each app's credential storage location);
2. Reassemble `settingsConfig` in the target app's structure — Claude / Codex / Gemini go through the existing `UniversalProvider::to_xxx_provider` conversion, while OpenCode / OpenClaw / Hermes construct the corresponding structure directly;
3. Preserve display metadata like name, notes, icon, icon color, and website;
4. Generate a new ID with a full UUID for each target app to avoid conflicts.

Same-app copy (e.g. Claude → Claude) clones the entire `settingsConfig` directly since the format is identical. Official / OAuth providers (without base_url and api_key) are rejected with a reason, avoiding invalid configs. The model name uses the target app's default; users can supplement the model list in the target app as needed.

### Provider config import/export

A new "Import/Export" button at the top of the provider list opens a dual-tab dialog:

- **Export**: multi-select providers (with select-all), then auto-download a `providers-{app}-{date}.json` file containing the version, export timestamp, app type, and provider list.
- **Import**: accepts a local JSON file or pasted JSON content. Import validates file size (10MB limit), JSON validity (frontend `JSON.parse` pre-check + backend serde parsing), version (only `1.0` accepted), and provider count (1000 limit). Imported providers get new IDs to avoid overwriting existing configs.

### Universal Provider app-level permission control

The `apps` field of Universal Provider is upgraded from a simple boolean to a configurable object supporting `enabled`:

- `enabled`: whether to sync to this app (same semantics as the original boolean).

For backward compatibility, the permission value uses an `#[serde(untagged)]` enum that supports both the old simple boolean (`true` / `false`) and the new detailed object (`{ enabled }`), with no migration needed for existing configs. The Rust side provides `is_enabled()` convenience methods to uniformly handle both forms, and all business logic using the `apps` field (deleting sub-providers, syncing to apps, etc.) has been switched to the new API.

## Changes

### Format conversion strategy for cross-app copy

The structural difference between apps' `settingsConfig` is the core challenge of cross-app copy. This release's strategy: use `base_url` + `api_key` as the intermediate credential representation across apps, resolve them from the source provider, then regenerate in the target structure. Claude / Codex / Gemini reuse the existing `UniversalProvider::to_xxx_provider` conversion logic (these methods were originally designed for "Universal Provider sync to apps"), while OpenCode / OpenClaw / Hermes construct the corresponding structure directly in the command layer (OpenCode's `options.{baseURL, apiKey}`, OpenClaw's flat `{baseUrl, apiKey, api}`, Hermes's flat snake_case `{base_url, api_key}`). This avoids introducing a new conversion abstraction and ensures output format consistency with the "Universal Provider sync" path.

### Security validation for import

The import path adds three layers of validation: a 10MB file size limit (checked both in the frontend file picker and the backend command), JSON validity (frontend `JSON.parse` pre-check for friendly messages, backend serde parsing as a backstop), and version + provider count (only `version: "1.0"` accepted, provider count ≤ 1000). Imported providers always get new IDs (with full UUIDs) and never overwrite existing same-named providers.

## Fixes

### i18n key type conflict

`provider.copyToApps` was previously defined as both a string (button title) and an object (dialog text), causing i18next to report "returned an object instead of string". Unified to an object structure; the button title now uses `provider.copyToApps.title`.

### Field name error in test mocks

The mock Provider objects in unit tests mistakenly used the snake_case `in_failover_queue` instead of the actual type's `inFailoverQueue`; corrected.

## Upgrade Notes

### Cross-app copy does not auto-fill the model list

Cross-app copy correctly fills in the API Key, Base URL, and display metadata (name / notes / icon), but the **model list uses the target app's default or is left empty**, because the field structure for model config differs significantly across apps. After copying, please supplement the models in the target app as needed.

### Official / OAuth providers do not support cross-app copy

Official and OAuth providers (e.g. Codex ChatGPT subscription, Gemini Google OAuth) have no extractable base_url and api_key, and cross-app copy rejects them with a reason. This is expected — such providers' credentials are managed by the app itself and cannot be shared across apps.

### Universal Provider permission field is backward-compatible

After the `apps` field upgrade, old simple boolean configs (`"claude": true`) still work with no migration needed. If you want to use the object form, change the boolean to an object (`"claude": { "enabled": true }`).

This release continues the risk notices for reverse-proxy features from previous versions.

**Claude Desktop third-party provider proxy switching**: when CC Switch's built-in proxy gateway routes Claude Desktop requests to a third-party provider, the same billing, compliance, and data retention constraints of the target provider apply.
