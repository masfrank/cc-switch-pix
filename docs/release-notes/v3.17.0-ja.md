# CC Switch v3.17.0

> このリリースの焦点は**複数アプリ間でプロバイダー設定を何度も入力し直す手間をなくすこと**——「モデルとプロバイダー」設定にアプリ間再利用機能を導入します。中核は三つ：第一に**アプリ間クイックコピー**、任意のプロバイダーカードからワンクリックで設定（API Key / Base URL / 名前 / メモ / アイコンなど）を他のアプリに配信し、コピー時に各アプリの設定フォーマット変換を自動完了します；第二に**プロバイダー設定のインポート/エクスポート**、選択したプロバイダーを JSON ファイルに一括エクスポート、またはファイル / クリップボードからインポートしてバックアップやクロスデバイス移行に対応します；第三に**汎用プロバイダーのアプリ権限制御**、従来の単純なブール値の有効フラグを `enabled` の設定可能なオブジェクトにアップグレードし、旧設定との完全な後方互換性を保ちます。

## 使い方ガイド

このリリースの新機能はプロバイダーリスト、汎用プロバイダーパネル、設定に分散しています。以下のシナリオと併せてお読みください：

- **アプリ間コピー**：任意のアプリのプロバイダーカードで、操作エリアに新設された「他のアプリにコピー」ボタン（共有アイコン）をクリックし、対象アプリを選択します。CC Switch はソースプロバイダーから API Key と Base URL を抽出し、対象アプリの設定構造（Claude の `env.ANTHROPIC_*`、Codex の `auth` + TOML `config`、Gemini の `env.GEMINI_*`、OpenCode の `options`、OpenClaw / Hermes のフラット構造）に従って再生成するため、アプリごとの再入力が不要になります。
- **インポート / エクスポート**：プロバイダーリスト上部に「インポート/エクスポート」ボタンを新設しました。エクスポートは複数プロバイダーを選択して日付入り JSON ファイルを自動ダウンロード、インポートはローカル JSON ファイルまたは内容の直接貼り付けに対応し、ファイルサイズ、JSON の妥当性、バージョンの検証を行います。
- **汎用プロバイダー権限**：汎用プロバイダーの `apps` フィールドがブール値からオブジェクトにアップグレードされ、各アプリの有効状態をきめ細かく設定でき、旧設定との完全な後方互換性を保ちます。

## 概要

CC Switch v3.17.0 は v3.16.5 に続く機能更新で、中核は**「モデルとプロバイダー」設定を特定のアプリから切り離し、アプリ間の設定効率と一貫性を向上させること**です。これまでは、同じモデルプロバイダー、同じモデルリストであっても、Claude Code、Codex、Gemini、OpenCode、OpenClaw、Hermes の各アプリごとに設定し直す必要がありました。本リリースはこの問題に対して二層のソリューションを提供します：**クイックコピー**（アプリ間設定クローン + フォーマット変換）と**インポート/エクスポート**（バックアップと移行）、さらに既存の**汎用プロバイダー**の権限次元を強化しています。

アプリ間コピーは単純なフィールドコピーではありません——各アプリの `settingsConfig` 構造は大きく異なり（Claude は `env.ANTHROPIC_BASE_URL` / `ANTHROPIC_AUTH_TOKEN`、Codex は `auth.OPENAI_API_KEY` と TOML `config` 文字列、Gemini は `env.GEMINI_*`、OpenCode は `options` 配下、OpenClaw と Hermes はフラットな camelCase / snake_case）、CC Switch はソースプロバイダーから `base_url` + `api_key` の中間証明情報ペアを解決し、対象アプリの構造に再組み立てします。同時に名前、メモ、アイコン、ウェブサイトなどの表示メタデータを保持します。公式 / OAuth プロバイダー（base_url と api_key がないもの）は理由とともに拒否され、不良設定の発生を防ぎます。

- **プロバイダーのアプリ間クイックコピー**：プロバイダーカードからワンクリックで設定を他のアプリに配信し、コピー時に API Key / Base URL のフォーマット変換を自動完了。全 7 アプリをカバーし、アプリごとの再入力を不要にします。

- **四言語 i18n 同期**：新機能の中国語 / 英語 / 日本語 / 繁体字中国語の翻訳をすべて完備。

## 新機能

### プロバイダーのアプリ間クイックコピー

これが本リリースのコア機能で、issue #4599 で報告された「同じプロバイダーをアプリごとに設定し直さなければならない」問題に対処します。任意のアプリのプロバイダーカードで、操作エリアに新設された「他のアプリにコピー」ボタン（Share2 アイコン）をクリックすると、アプリ選択ダイアログが開き、全選択または対象アプリをチェックできます。CC Switch は：

1. ソースプロバイダーから `base_url` + `api_key` を解決（各アプリの証明情報格納位置をカバーする既存の `resolve_usage_credentials` を再利用）；
2. 対象アプリの設定構造に従って `settingsConfig` を再組み立て——Claude / Codex / Gemini は既存の `UniversalProvider::to_xxx_provider` 変換を使用し、OpenCode / OpenClaw / Hermes は対応する構造を直接構築；
3. 名前、メモ、アイコン、アイコン色、ウェブサイトなどの表示メタデータを保持；
4. 各対象アプリに完全な UUID を持つ新 ID を生成し、競合を回避。

同じアプリへのコピー（例：Claude → Claude）は構造が同一なため `settingsConfig` 全体を直接クローンします。公式 / OAuth プロバイダー（base_url と api_key がないもの）は理由とともに拒否され、無効な設定の発生を防ぎます。モデル名は対象アプリのデフォルト値を使用し、ユーザーは必要に応じて対象アプリでモデルリストを補足できます。

### プロバイダー設定のインポート/エクスポート

プロバイダーリスト上部に「インポート/エクスポート」ボタンを新設し、ツータブのダイアログを開きます：

- **エクスポート**：複数プロバイダーを選択（全選択対応）し、エクスポートをクリックすると `providers-{アプリ}-{日付}.json` を自動ダウンロード。ファイルにはバージョン、エクスポートタイムスタンプ、アプリタイプ、プロバイダーリストが含まれます。
- **インポート**：ローカル JSON ファイルまたは JSON 内容の直接貼り付けに対応。インポート時にファイルサイズ（10MB 上限）、JSON の妥当性（フロントエンド `JSON.parse` 事前チェック + バックエンド serde 解析）、バージョン（`1.0` のみ受け付け）、プロバイダー数（1000 上限）を検証し、インポートしたプロバイダーには新 ID を生成して既存設定の上書きを回避します。

### 汎用プロバイダーのアプリ権限制御

汎用プロバイダー（Universal Provider）の `apps` フィールドを単純なブール値から設定可能なオブジェクトにアップグレードし、`enabled` フィールドをサポートします：

- `enabled`：このアプリへの同期を有効にするか（元のブール値と同じ意味）。

後方互換性のため、権限値は `#[serde(untagged)]` 列挙型を使用し、旧来の単純なブール値（`true` / `false`）と新しい詳細オブジェクト（`{ enabled }`）の両方をサポートします。既存設定の移行は不要です。Rust 側は `is_enabled()` の便利メソッドを提供して両形態を統一処理し、`apps` フィールドを使用する全ビジネスロジック（サブプロバイダーの削除、各アプリへの同期など）を新 API に切り替えました。

## 変更

### アプリ間コピーのフォーマット変換戦略

各アプリの `settingsConfig` 構造の違いがアプリ間コピーの中核の課題です。本リリースの戦略：`base_url` + `api_key` をアプリ間の中間証明情報表現として使用し、ソースプロバイダーから解決した上で対象構造に再生成します。Claude / Codex / Gemini は既存の `UniversalProvider::to_xxx_provider` 変換ロジック（これらのメソッドは元々「汎用プロバイダーの各アプリへの同期」のために設計されたもの）を再利用し、OpenCode / OpenClaw / Hermes はコマンド層で対応する構造を直接構築します（OpenCode の `options.{baseURL, apiKey}`、OpenClaw のフラットな `{baseUrl, apiKey, api}`、Hermes のフラットな snake_case `{base_url, api_key}`）。これにより新しい変換抽象を導入せず、「汎用プロバイダーの同期」パスと一致する出力フォーマットを保証します。

### インポートのセキュリティ検証

インポートパスには三層の検証を追加しました：10MB のファイルサイズ上限（フロントエンドのファイル選択とバックエンドコマンドで二重チェック）、JSON の妥当性（フロントエンド `JSON.parse` 事前チェックで分かりやすいメッセージ、バックエンド serde 解析が最終防衛）、バージョンとプロバイダー数（`version: "1.0"` のみ受け付け、プロバイダー数 ≤ 1000）。インポートしたプロバイダーは常に新 ID（完全な UUID 付き）を取得し、同名の既存プロバイダーを上書きすることはありません。

## 修正

### i18n キーの型競合

`provider.copyToApps` が文字列（ボタン title）とオブジェクト（ダイアログ文案）の両方として定義されており、i18next が「returned an object instead of string」を報告していました。オブジェクト構造に統一し、ボタン title は `provider.copyToApps.title` を使用するよう修正しました。

### テストモックのフィールド名エラー

ユニットテストのモック Provider オブジェクトが誤って snake_case の `in_failover_queue` を使用しており、実際の型の `inFailoverQueue` と不一致だったため修正しました。

## アップグレードの注意

### アプリ間コピーはモデルリストを自動入力しません

アプリ間コピーは API Key、Base URL、表示メタデータ（名前 / メモ / アイコン）を正しく入力しますが、**モデルリストは対象アプリのデフォルト値または空になります**。モデル設定のフィールド構造はアプリ間で大きく異なるためです。コピー後、必要に応じて対象アプリでモデルを補足してください。

### 公式 / OAuth プロバイダーはアプリ間コピーに対応しません

公式および OAuth プロバイダー（例：Codex ChatGPT サブスクリプション、Gemini Google OAuth）には抽出可能な base_url と api_key がなく、アプリ間コピーは理由とともに拒否されます。これは想定された動作です——这类プロバイダーの証明情報は対応するアプリ自身が管理しており、アプリ間で共有できません。

### 汎用プロバイダー権限フィールドの後方互換性

`apps` フィールドのアップグレード後、旧来の単純なブール値設定（`"claude": true`）は引き続き有効で、移行は不要です。オブジェクト形式を使用したい場合は、ブール値をオブジェクト形式（`"claude": { "enabled": true }`）に変更してください。

## リスク通知

本リリースは前バージョンのリバースプロキシ系機能に対するリスク通知を引き継ぎます。

**Codex OAuth リバースプロキシ**：ChatGPT サブスクリプションを使用した Codex OAuth リバースプロキシは OpenAI の利用規約に違反する可能性があります。詳細は [v3.13.0 リリースノート](v3.13.0-ja.md#-リスク通知) を参照してください。

**Codex サードパーティプロバイダーの Chat ルーティング**：CC Switch のローカルプロキシが Codex リクエストを変換してサードパーティプロバイダーに転送する際、各プロバイダーの課金、コンプライアンス、データ保持の制約が異なります。使用前に対象プロバイダーの利用規約をお読みください。

**Claude Desktop サードパーティプロバイダーのプロキシ切り替え**：CC Switch の組み込みプロキシゲートウェイが Claude Desktop のリクエストをサードパーティプロバイダーに転送する際も、対象プロバイダーの課金、コンプライアンス、データ保持の制約が適用されます。

ユーザーがこれらの機能を有効にすることは、関連リスクを自ら負うことに同意したものとみなされます。CC Switch はこれらの機能の使用に起因するアカウント制限、警告、サービス停止について責任を負いません。
