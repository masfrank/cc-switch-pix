# CC Switch v3.17.0

> 这一版的重点是**解决多应用间供应商配置重复录入的痛点**——为「模型与供应商」配置引入跨应用复用能力。核心有三：一是**跨应用快捷复制**，在任意供应商卡片上一键把配置（API Key / Base URL / 名称 / 备注 / 图标等）分发到其他应用，并在复制时自动完成各应用的配置格式转换；二是**供应商配置的导入导出**，支持把选中的供应商批量导出为 JSON 文件、或从文件 / 剪贴板导入回来，方便备份与跨设备迁移；三是**统一供应商的应用权限管控**，把原来的简单布尔启用状态升级为可配置 `enabled` 的对象形式，并完全向后兼容旧配置。

## 使用攻略

本版的新能力分布在供应商列表、统一供应商面板与设置中，建议结合以下场景了解：

- **跨应用复制**：在某个应用的供应商卡片上点击新的「复制到其他应用」按钮（分享图标），勾选目标应用即可。CC Switch 会自动从源供应商提取 API Key 与 Base URL，再按目标应用的配置结构（Claude 的 `env.ANTHROPIC_*`、Codex 的 `auth` + TOML `config`、Gemini 的 `env.GEMINI_*`、OpenCode 的 `options`、OpenClaw / Hermes 的扁平结构）重新生成配置，不再需要每个应用逐一录入。
- **导入 / 导出**：供应商列表顶部新增「导入 / 导出」按钮。导出时可多选供应商、自动下载为带日期的 JSON 文件；导入时支持选择本地 JSON 文件或直接粘贴内容，并对文件大小、JSON 合法性与版本号做了校验。
- **统一供应商权限**：统一供应商的 `apps` 字段从布尔升级为对象，可细粒度配置每个应用的启用状态，并完全向后兼容旧配置。

## 概览

CC Switch v3.17.0 是 v3.16.5 之后的一版功能更新，核心是**把「模型与供应商」配置从具体应用中解耦、提升多应用间的配置效率与一致性**。在此之前，即使是同一个模型供应商、同样的模型列表，用户也需要针对 Claude Code、Codex、Gemini、OpenCode、OpenClaw、Hermes 每个应用各配一遍。本版围绕这个问题提供两层方案：**快捷复制**（跨应用配置克隆 + 格式转换）与**导入导出**（备份与迁移），并对既有的**统一供应商**做了权限维度的增强。

跨应用复制不是简单的字段拷贝——各应用的 `settingsConfig` 结构差异很大（Claude 用 `env.ANTHROPIC_BASE_URL` / `ANTHROPIC_AUTH_TOKEN`、Codex 用 `auth.OPENAI_API_KEY` 加 TOML `config` 字符串、Gemini 用 `env.GEMINI_*`、OpenCode 嵌套在 `options` 下、OpenClaw 与 Hermes 是扁平的 camelCase / snake_case），CC Switch 会从源供应商解析出 `base_url` + `api_key` 这组中间凭证，再按目标应用的结构重新组装，同时保留名称、备注、图标、官网链接等元数据。官方 / OAuth 类供应商（无 base_url 与 api_key）会被拒绝复制并给出原因，避免产生坏配置。

**发布日期**：2026-07-02

- **供应商配置导入导出**：供应商列表新增「导入 / 导出」入口，支持多选导出为带日期的 JSON 文件、从文件或剪贴板导入，并对文件大小、JSON 合法性、版本号与供应商数量做了校验。

- **四语国际化同步**：新增功能的中 / 英 / 日 / 繁中翻译全部补齐。

## 新功能

### 跨应用快捷复制供应商

这是本版的核心功能，针对 issue #4599 反馈的「同一供应商要在每个应用各配一遍」问题。在任意应用的供应商卡片上，点击操作区新增的「复制到其他应用」按钮（Share2 图标），会弹出一个应用选择对话框，可全选或勾选目标应用。CC Switch 会：

1. 从源供应商解析出 `base_url` + `api_key`（复用已有的 `resolve_usage_credentials`，覆盖各应用的凭证存放位置）；
2. 按目标应用的配置结构重新组装 `settingsConfig`——Claude / Codex / Gemini 走既有的 `UniversalProvider::to_xxx_provider` 转换，OpenCode / OpenClaw / Hermes 则直接构造对应结构；
3. 保留名称、备注、图标、图标颜色、官网链接等展示元数据；
4. 为每个目标应用生成带完整 UUID 的新 ID，避免冲突。

同应用复制（如 Claude → Claude）会直接克隆整个 `settingsConfig`，因为格式一致。官方 / OAuth 供应商（无 base_url 与 api_key）会被拒绝复制并提示原因，避免产生无效配置。模型名称会用目标应用的默认值，用户可在目标应用中按需补充模型列表。

### 供应商配置导入导出

供应商列表顶部新增「导入 / 导出」按钮，打开一个双 Tab 对话框：

- **导出**：可多选供应商（支持全选），点击导出后自动下载为 `providers-{应用}-{日期}.json`，文件内含版本号、导出时间戳、应用类型与供应商列表。
- **导入**：支持选择本地 JSON 文件，或直接在文本框粘贴 JSON 内容。导入时会校验文件大小（10MB 上限）、JSON 合法性（前端 `JSON.parse` 预校验 + 后端 serde 解析）、版本号（仅接受 `1.0`）与供应商数量（1000 上限），导入的供应商会生成新 ID 避免覆盖已有配置。

### 统一供应商应用权限管控

统一供应商（Universal Provider）的 `apps` 字段从简单布尔升级为可配置对象，支持 `enabled` 字段：

- `enabled`：是否启用同步到该应用（与原布尔值语义一致）。

为保持向后兼容，权限值用 `#[serde(untagged)]` 枚举表示，既支持旧的简单布尔值（`true` / `false`），也支持新的详细对象（`{ enabled }`），现有配置无需迁移。Rust 端提供 `is_enabled()` 便捷方法统一处理两种形态，所有使用 `apps` 字段的业务逻辑（删除子供应商、同步到各应用等）已切换到新 API。

## 变更

### 跨应用复制的格式转换策略

不同应用的 `settingsConfig` 结构差异是跨应用复制的核心难点。本版采用的策略是：以 `base_url` + `api_key` 作为跨应用的中间凭证表示，从源供应商解析出来后按目标结构重新生成。Claude / Codex / Gemini 复用既有的 `UniversalProvider::to_xxx_provider` 转换逻辑（这些方法本就为「统一供应商同步到各应用」设计），OpenCode / OpenClaw / Hermes 则在命令层直接构造对应结构（OpenCode 的 `options.{baseURL, apiKey}`、OpenClaw 的扁平 `{baseUrl, apiKey, api}`、Hermes 的扁平 snake_case `{base_url, api_key}`）。这样既不引入新的转换抽象，也保证了与「统一供应商同步」路径一致的输出格式。

### 导入文件的安全校验

导入路径加了三道校验：文件大小上限 10MB（前端文件选择与后端命令双重检查）、JSON 合法性（前端 `JSON.parse` 预校验给出友好提示，后端 serde 解析兜底）、版本号与供应商数量（仅接受 `version: "1.0"`、供应商数 ≤ 1000）。导入的供应商一律生成新 ID（带完整 UUID），不会覆盖已有同名供应商。

## 修复

### i18n 键值类型冲突

`provider.copyToApps` 此前同时被定义为字符串（按钮 title）和对象（对话框文案），导致 i18next 报「returned an object instead of string」。已统一为对象结构，按钮 title 改用 `provider.copyToApps.title`。

### 测试 mock 的字段名错误

单元测试里的 mock Provider 对象误用了 snake_case 的 `in_failover_queue`，与实际类型的 `inFailoverQueue` 不符，已订正。

## 升级提醒

### 跨应用复制不会自动填充模型列表

跨应用复制会正确填入 API Key、Base URL 与展示元数据（名称 / 备注 / 图标），但**模型列表会用目标应用的默认值或留空**，因为不同应用对模型配置的字段结构差异较大。复制后请在目标应用中按需补充模型。

### 官方 / OAuth 供应商不支持跨应用复制

官方与 OAuth 类供应商（如 Codex ChatGPT 订阅、Gemini Google OAuth）没有可提取的 base_url 与 api_key，跨应用复制会被拒绝并提示原因。这是预期行为——这类供应商的凭证由对应应用自身管理，无法跨应用共享。

### 统一供应商权限字段向后兼容

统一供应商的 `apps` 字段升级后，旧的简单布尔值配置（`"claude": true`）仍然有效，无需迁移。若你需要使用对象形式，可将布尔值改为对象（`"claude": { "enabled": true }`）。

## 风险提示

本版本继续沿用此前版本对反向代理类功能的风险提示。

**Codex OAuth 反向代理**：使用 ChatGPT 订阅的 Codex OAuth 反代可能违反 OpenAI 服务条款，详情见 [v3.13.0 release notes](v3.13.0-zh.md#️-风险提示)。

**Codex 第三方供应商 Chat 路由**：通过 CC Switch 本地代理把 Codex 请求转换并转发到第三方供应商时，各供应商对计费、合规与数据留存的约束不同，请在使用前阅读目标供应商的服务条款。

**Claude Desktop 第三方供应商代理切换**：通过 CC Switch 内置代理网关把 Claude Desktop 的请求转到第三方供应商时，同样需要遵守目标供应商的计费、合规与数据留存约束。

用户启用上述功能即表示自行承担相关风险。CC Switch 不对因使用这些功能而导致的任何账号限制、警告或服务暂停承担责任。
